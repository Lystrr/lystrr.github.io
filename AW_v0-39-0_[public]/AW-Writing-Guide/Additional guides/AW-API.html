Accidental Woman API v0.7

Character Variables

A lot of information is stored about each character in Accidental Woman, including the player. In many cases, there is little difference between PC and NPC data, and so the reference is combined. An example is the Body Class, which is identical for all characters. Another important piece of information is that each character's data is stored as a class object. This means that it functions a bit like a tree, with all the information stored inside a single property. For the player, it is ↂ.pc. For NPCs, they use their npcid instead of "pc" as the name of their property. For example, this is Lily: aw.npc.n101. npcid is a code that is generated for each npc. More prominent characters (like Lily) have a smaller code. The real estate agent Tiffany is "n1001" and the pharmacist at Bullseye is "n1002".

Technically, the variable is the main item that contains the object, and everything inside is called properties. Sort of like variable.property.property...property The terminology isn't terribly important, but it is important to know that when accessing these properties, you have to provide the names of the parent properties as well. It's sort of like working your way to the data you want. As an example, reading the number of balls a character has would be accomplished like this: balls = aw.npc.n101.body.balls.count

The information about variables has been laid out in what I hope is an easy-to-use format. Properties are contained inside an indented "box". At the top of the box, the main information about it is listed. In white is the proper name of the variable (with correct capitalization). Next in pale yellow is the data type that property contains. You can learn all about JavaScript data types online, such as this reference at W3 Schools. For arrays, this API uses a little TypeScript notation for additional information. Instead of simply listing "array", it will list another data type followed by brackets (ex: string[] is an array containing strings). After the data type, you may see further information in blue or green. This is information about allowed values. For numbers it will likely be a range such as " 1 to 10", while for strings you may see a bunch of words. If the text is green, it is indicating that the property is a special sub-class, such as the Tits Class, or that the property is what is known as a "getter" . Getters return values, often ones calculated based on other values, and do not actually store a value. Below the basic information about the property, there will be a smaller black box with information about it. Usually this describes what the property is for, what unit it's in, or what the values mean.

Below is an example of an object in this format, with some extra info about how the API format works.

alpha object

This is an example of a property containing an object. An object has further properties inside it, which are listed below in a nested fashion.

.beta number 0 to 100

properties in an object will start with a period (".") as a reminder that they are inside their parent property. This means to access this 'beta' property, you'd use parent.alpha.beta instead of just parent.beta.

.delta string blue, red, green, yellow

With strings, you may see a list of values like these colors. In this case, it means that those values are valid. In most cases, strings other than the ones listed are not valid. The main exception is in arrays of strings (string[]) like the common tag property.

.gamma object

sometimes a property in an object will also contain an object! In this case, it's properties will be further nested like this.

gamma.omega boolean true or false

another level of nesting will be indicated by having the parent property's name before it, just to further distinguish things. In this case, to access "omega" you would use parent.alpha.gamma.omega

Accidental Woman Parser API

The Accidental Woman Parser is a single function that takes a provided command (and arguments, if used) and returns a word, sentence fragment, or entire sentence. The parser has two modes of operation: targeted and untargeted. Targeted operation uses the variables from the target when parsing, to return words that are a good fit for the character. Untargeted mode is basically just returning a word based on either average attributes, or an attribute supplied with an argument. To make use easy, parser calls are split into three macros.

The macro key is either "<<p" for the player, "<<n" for a npc, or "<<w" for a word that isn't based on specific character. (Note that the word variant doesn't work with every command, and is only used for certain specific purposes.)

Using <<n requires supplying either the npcid, or npc index afterward. Examples: index <<n 3 ass.size>> or npcid <<n n101 ass.size>>

Using <<w requires a value be provided after the parse command. Otherwise, it won't know the value to base the response on. Example: <<w ass.size 5>>

Qualitative Functions

Returns a word associated with a size or other scale-type value (quality) of the command word. The qualifier for these functions is [.qual .size .q .s] (Example: ass.qual returns a word for the size of the target's ass.)

KEY:
Unless noted in the specific command, you can provide a mod value for "p" and "n" parser macros. At the end of the call, type "mod" followed by the mod value. This will add the mod value to the character's normal value. (use a negative number to subtract.) Example: <<p ass.size "mod" 1>> will make the ass 1 size larger when choosing a size word.

Simple Noun Functions

Simple noun functions return a basic noun by itself. These are for use by composite functions when creating a subject, or by themselves anytime a noun needs to be used by itself. Each function contains a list of weighted words ranging from 10 to 30 variations depending on the subject.

The nouns are typically relevant to the target, to match the target linguistically. For example the vulva function (for external female genitalia) won't return "bearded clam" if the target has no pubic hair there. Similarly, it's more likely to use a word like "jugs" for large breasts, but would never use it for small breasts. The untargeted word version (<<w) uses a mid-range value. It will never display words only appropriate for extreme values, and thus is somewhat safe to use. Using a target is still preferred though unless referring to a concept (I <3 tits) rather than a specific character (she grabs her tits).

The other form of relevance is related to the player themselves, for specific nouns where this makes more sense than being based exclusively on a character. In this case, a character kink or trait of the player character can expand or contract the word selection list. For example, someone with the pregnancy or risky sex kinks will see more nouns tied to pregnancy/fertility, while characters without them will not see those nouns. In some cases, such as with penis or testicle size, the player character's kinks can shift the thresholds for size categories. A size queen, for example, will have nouns returned as if the equipment is smaller.
